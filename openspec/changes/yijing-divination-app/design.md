## Context

本项目是全新的个人易经占卜 Web 应用（纯前端）。设计概要已记录在 `DESIGN.md` 中。Phase A 实现周易古占：本卦/变卦/动爻计算、64 卦数据查询、DeepSeek AI 结构化解读、结果反馈追踪、统计分析。

### 约束

- 纯前端，无后端服务
- 规则引擎不可学习（算法永不修改）
- API Key 存 localStorage，永不离开浏览器
- 数据存 IndexedDB，JSON 导出备份

## Goals / Non-Goals

**Goals:**
- 构建严格遵循周易古占规则的卦象计算引擎
- 集成 DeepSeek API，双调用架构实现高质量解读
- 建立完整的"问题→卦象→解读→反馈→统计"数据闭环
- 实验派：累计数据供长期验证
- 将每次占问作为可复盘的个人决策记录，保存占问前预判、AI 判断、事后结果和可验证判断点

**Non-Goals:**
- Phase A 不做六爻纳甲（互卦/错卦/综卦/六亲/六神等）
- 不实现多用户系统
- 不做草稿恢复（摇卦中途退出不保存）
- 不做移动端原生 App（Web 应用即可）

## Decisions

### D1: Double-Call AI 架构

**选择**：两次 API 调用——第一次结构化推理（输出 JSON），第二次受控叙事（输出展示文本）。

**理由**：
- 一次调用无法同时满足"结构化可解析输出"和"高质量中文叙事"
- 第一次调用严格约束 JSON 格式，保证趋势判断 / 条件 / 时间窗口字段可解析
- 第二次调用将结构转为自然语言，施加内容约束（禁语、引用出处）
- 避免 AI "一次做太多事"导致的输出质量下降

**替代方案**：单次调用（prompt 同时要求结构和叙事）——测试表明输出不稳定，JSON 格式频繁断裂或漏字段。

### D2: 规则引擎纯函数设计

**选择**：起卦计算、卦象查询、爻辞检索全部用纯函数实现，无副作用，无状态。

**理由**：
- 纯函数可确定性测试（TDD 最适用）
- 规则层不可学习的保证——没有可修改的参数或权重
- 与 UI 完全解耦，未来可在任何框架中复用

### D3: IndexedDB 而非 localStorage 作为主存储

**选择**：使用 IndexedDB 存储占卜记录，localStorage 仅存 API Key。

**理由**：
- 占卜记录字段多、体积大（含 AI 完整响应文本），localStorage 有 5MB 限制
- IndexedDB 支持索引查询（按类别、时间、反馈状态），统计面板需要这些查询能力

### D4: crypto.getRandomValues() 作为虚拟摇卦随机源

**选择**：使用 Web Crypto API 而非 Math.random()，并模拟三枚铜钱分别落面，而不是均匀生成 6/7/8/9。

**理由**：
- crypto.getRandomValues() 是密码学安全的伪随机数生成器
- 虽然用户看不见差异，但这是"尽可能随机"的姿态
- 传统认为起卦的随机质量影响卦象的有效性
- 三枚铜钱的爻值概率不是均匀分布：6=1/8，7=3/8，8=3/8，9=1/8；均匀生成会改变传统规则

### D5: 背面计数法手动输入

**选择**：手动输入时用户选背面数（0/1/2/3），系统自动判断爻型。

**理由**：
- 用户无需记住 6/7/8/9 的对应关系
- 背面数是现实中真实的观察结果，不需要二次转换
- 减少输入错误率

### D6: 数据结构版本化

**选择**：占卜记录保存 `schemaVersion`，导出 JSON 使用 `{ app, schemaVersion, exportedAt, records }` 顶层结构；记录内时间统一存 ISO 字符串。

**理由**：
- 后续 Phase B 一定会扩展字段，版本号是导入旧数据和迁移的最低成本保障
- JSON 导出时 `Date` 会变成字符串，直接用 ISO 字符串可避免运行时和备份文件的类型不一致
- 顶层元数据可以防止误导入其他 JSON 文件，并为未来批量迁移留出口

### D7: AI 结果可追踪

**选择**：每次 AI 解读保存为 `interpretations[]`，默认解读和深度分析并存；每条保存 `model`、`promptVersion`、`temperature`、`type`、原始响应和可验证 `claims`。

**理由**：
- 统计 pro vs flash 准确率时，需要把反馈和模型版本关联起来
- Prompt 会迭代，历史数据必须知道由哪个 prompt 产生
- `claims` 将自然语言解读拆成可复盘判断点，避免反馈只能粗糙地评价整篇文章

### D8: 反馈延迟与稍后提醒

**选择**：新记录默认 `feedback.status = pending`，但只有 `feedback.dueAt <= now` 时才主动提醒；用户可选择稍后提醒。

**理由**：
- 刚占问完成就要求反馈，会产生无效反馈
- 不同类型问题需要不同观察窗口，`dueAt` 比简单 pending 更符合真实复盘节奏
- 稍后提醒降低打断感，同时保留数据闭环

### D9: 运行时 schema 校验

**选择**：AI JSON、导入 JSON、IndexedDB 记录都需要运行时 schema 校验。

**理由**：
- TypeScript 只约束编译期，无法保证模型输出和用户导入文件正确
- JSON.parse 成功不代表字段合法
- 早期明确失败原因，比把坏数据写入历史记录更安全

### D10: 组件层次

**选择**：
```
AppShell → 路由
  ├─ HomeView (入口 + 欢迎)
  ├─ DivineView (起卦页)
  │   ├─ QuestionInput
  │   ├─ CategoryPicker
  │   ├─ MethodToggle (虚拟/手动)
  │   ├─ VirtualCoins (CSS 2D 翻转动画)
  │   ├─ ManualInput (背面数选择)
  │   └─ HexagramBoard (卦盘逐爻展示)
  ├─ ResultView (解读结果)
  │   ├─ Interpretation (结构化展示)
  │   └─ DeepAnalysis (pro 模型深度分析)
  ├─ HistoryView (历史列表)
  ├─ HistoryDetailView (单条详情)
  ├─ StatsView (统计面板)
  └─ SettingsView (API Key + 导出/导入)
```

每个 view 是独立路由页面，组件按功能域分组。

## Risks / Trade-offs

- **[R] DeepSeek API 不可用** → 纯前端无后端回退，解读功能完全失效。缓解：显示规则引擎的原始结果（卦名、卦辞、爻辞），用户可自行查阅古籍。
- **[R] IndexedDB 数据丢失（清浏览器缓存）** → 缓解：定期 JSON 导出提醒 + 导出文件独立保存。
- **[R] 前端 API Key 暴露在请求中** → 个人工具，浏览器 Network 面板可看到，风险可接受。如未来需要安全性，可加后端代理。
- **[R] Double-call 延迟** → 两次 API 调用增加 ~3-5 秒等待。缓解：显示推理进度指示器。
- **[R] AI 解读质量波动** → DeepSeek 模型可能在不同版本间质量变化。缓解：Prompt 版本管理 + 用户反馈即质量信号。
- **[R] 浏览器直连 DeepSeek 可能受 CORS 限制** → Phase A 纯前端目标下必须先做最小 API 连通性验证；若不可用，需要后端代理或手动复制解读作为替代方案。
- **[R] 小样本统计误导** → 准确率展示必须同时显示样本量；误判模式分析仅在已反馈样本足够时展示。
- **[R] 64 卦映射错位** → 通过固定映射表和多个已知卦例测试锁定，不依赖隐含算法解释。

## 一、项目总览

**项目名称**：易经占卜

**定位**：周易古占 · 实验派个人决策辅助系统

**核心理念**：
- 规则引擎不可学习——卦象计算算法永不修改
- 一事不二占（宽松模式：警告 + 允许跳过）
- 每次占卜保留三类信息：占前预判 → 占中卦象 → 占后反馈

**技术栈**：

| 类别 | 技术 |
|------|------|
| 框架 | React 19 + TypeScript 6.0 |
| 构建 | Vite 8 |
| 样式 | Tailwind CSS v4 + Nothing Design 设计系统 |
| 动画 | GSAP 3 + ScrollTrigger + @gsap/react |
| 路由 | react-router-dom 7 |
| 后端 | Supabase（PostgreSQL + Auth + RLS）|
| AI | DeepSeek API（v4-flash / v4-pro）|
| 校验 | Zod 4 |
| 测试 | Vitest + Playwright |
| 部署 | Vercel 静态部署 |

**文件结构**：

```
src/
├── ai/           # AI 解读层（6 文件：client, double-call, reasoning, narrative, prompt-builder, fallback）
├── auth/         # 认证层（4 文件：Context, Login, Register, ProtectedRoute）
├── components/
│   ├── casting/  # 起卦组件（5 文件：VirtualCoins, ManualInput, HexagramBoard, RitualIntro, ClosingRitual）
│   ├── feedback/ # 反馈组件（2 文件：FeedbackForm, FeedbackList）
│   ├── layout/   # 布局（AppShell）
│   ├── result/   # 结果展示（2 文件：Interpretation, AIProgressIndicator）
│   └── ui/       # UI 基础组件（6 文件：Button, GlassCard, Input, Tag, StepIndicator, PageTransition）
├── data/         # 古籍数据（hexagrams.json 64卦完整数据, najia-bagua.json）
├── db/           # 数据层（records.ts, export-import.ts）
├── engine/       # 规则引擎（12 文件：casting, hexagram-*, trigram-wuxing, calendar, najia, yarrow-stalk, duplicate-check, feedback-analytics, user-profile, prompt-optimizer）
├── hooks/        # 自定义 Hooks（6 文件）
├── lib/          # 工具库（7 文件）
├── pages/        # 页面视图（7 文件）
└── types/        # 类型定义
```

## 二、前端页面逻辑

### 路由结构（7 个路由）

| 路由 | 页面 | 核心功能 |
|------|------|---------|
| `/login` | 登录 | Supabase Auth 用户名+密码 |
| `/register` | 注册 | 用户名+密码，≥6位 |
| `/` | 首页 | 今日统计、待反馈列表、CTA 按钮、快捷导航 |
| `/divine` | 起卦页 | 四步引导流程 |
| `/result/:id` | 结果页 | 卦象展示、AI 解读、反馈表单 |
| `/history` | 历史记录 | 分类筛选、GSAP 动画列表 |
| `/history/:id` | 记录详情 | 完整记录回溯 + 反馈表单 |
| `/stats` | 统计面板 | 基础计数统计 |
| `/settings` | 设置 | API Key、导入导出、版本信息、退出登录 |

### 起卦页四步流程（`DivineView.tsx`）

| 步骤 | 内容 | 状态管理 |
|:--:|------|---------|
| 1 | **输入问题**：textarea（100字上限）+ 分类标签（工作/人际/财务/健康/其他）| `useDivination` hook |
| 2 | **记录预判**（可选）：预期/信心度(1-5)/计划行动 | `BeforeDivination` 类型 |
| 3 | **选择方式**：虚拟摇卦 / 手动输入 | `CastingMethod` 类型 |
| 4 | **起卦交互**：逐爻掷出或手动选背面数 | 6个 `LineValue` 组成卦象 |

每次步骤切换使用 `PageTransition` 组件做 GSAP 方向滑入/滑出动画。步骤进度由 `StepIndicator` 显示。

### 结果页逻辑（`ResultView.tsx`）

页面的自动行为：当记录加载完成且有 API Key 但无解读时，**自动触发** `triggerDefault()`。之后用户可手动触发 `triggerDeep()`（深度分析）。

### 历史记录（`HistoryView.tsx`）

支持分类筛选，列表项展示问题、日期、分类、反馈状态徽章。使用 GSAP stagger 入场动画。空状态显示"暂无记录"。

## 三、前端动画体系

动画引擎：**GSAP 3 + ScrollTrigger + @gsap/react**。全局通过 `src/lib/gsap.ts` 统一配置。

### 动画清单

| 位置 | 动画内容 | 技术细节 |
|------|---------|---------|
| `VirtualCoins.tsx` | 铜钱翻转 | `rotationY: 360°` + `elastic.out` 弹性回弹。3 枚铜钱错开 0.08s。动画完成后 `onComplete` 回调掷出结果并自动确认（600ms 延迟）|
| `HexagramBoard.tsx` | 爻逐条入场 | `opacity: 0, x: -24` → `opacity: 1, x: 0`，每爻 `power3.out` 0.35s |
| `HexagramBoard.tsx` | 卦名浮现 | `opacity: 0, y: 8, scale: 0.95` → 0.6s fade + 0.3s delay |
| `HexagramBoard.tsx` | 爻持续呼吸 | 阴爻微旋转 (repeat: -1, yoyo: true)，阳爻微缩放，动爻轻闪烁 |
| `PageTransition.tsx` | 页面切换 | 四方向（left/right/up/down）滑入滑出。入场 `power2.out` 0.5s，出场 `power2.in` 0.35s |
| `StepIndicator.tsx` | 步骤切换 | 当前步骤 dot 从 0.8 弹跳到 1.2（`back.out(1.7)`，repeat:1）|
| `HistoryView.tsx` | 列表入场 | Stagger `opacity` + `x` translation |
| `StatsView.tsx` | 统计卡片 | ScrollTrigger 触发入场 + 数字 counter 动画 |
| `RitualIntro.tsx` | 开卦呼吸引导 | 球体缩放动画 3 轮 ×14s（4s吸气→4s屏息→6s呼气）|
| `ClosingRitual.tsx` | 谢卦仪式 | fade-in overlay 模态（**未接入任何页面**）|
| `FeedbackList.tsx` | 反馈模态 | GSAP 入场动画 |
| `AIProgressIndicator.tsx` | AI 进度 | GSAP 旋转 spinner |

### 无障碍设计

- 所有动画组件通过 `useReducedMotion()` hook 检测 `prefers-reduced-motion`
- 开启后：GSAP animation 跳至 `progress(1)` 或使用 `duration: 0`
- GSAP `matchMedia` 全局配置：reduced motion → 全局暂停；mobile → duration 0.4s；desktop → duration 0.6s
- 提供 `useAccessibleAnimation()` hook 作为通用降级包装

## 四、卜卦方式方法

### 4.1 已实现的起卦方式

| 方式 | 概率分布 | 实现 | UI |
|------|---------|------|:--:|
| **虚拟铜钱法** | 6:12.5%, 7:37.5%, 8:37.5%, 9:12.5% | `casting.ts` → `tossThreeCoins()` → `crypto.getRandomValues()` | ✅ 完整的 GSAP 翻转动画 |
| **手动输入法** | 用户自选 0/1/2/3 背 | `ManualInput.tsx` → `tossResultToLineValue()` | ✅ 4宫格按钮 + 进度条 |
| **大衍筮法** | 6:6.25%, 7:31.25%, 8:43.75%, 9:18.75% | `yarrow-stalk.ts` → `castYarrowLine()` | ❌ 无 UI 入口 |

### 4.2 铜钱约定

```
字面（汉字）= 2
背面（花纹）= 3

三枚掷出算背面数：
  3 背 → 9 老阳 ⚊○ → 变阴
  2 背 → 8 少阴 ⚋   → 不变
  1 背 → 7 少阳 ⚊   → 不变
  0 背 → 6 老阴 ⚋× → 变阳
```

### 4.3 概率差异对比

| 爻值 | 铜钱法 | 大衍筮法 | 差异 |
|------|:---:|:---:|------|
| 6 老阴 | 12.5% | 6.25% | 铜钱法老阴多一倍 |
| 7 少阳 | 37.5% | 31.25% | 偏多 |
| 8 少阴 | 37.5% | 43.75% | 偏少 |
| 9 老阳 | 12.5% | 18.75% | 偏少 |

### 4.4 起卦交互细节

- 虚拟摇卦：用户手动点击 6 次，每次掷一枚爻（初爻→上爻）
- **第一次点击铜钱的时间** 记录为 `castingTimestamp`
- 动画完成后自动确认（600ms），无需额外点击
- 结果展示：X 背 Y 字 → 爻值 → 爻名（如 `2 背 1 字 → 8 少阴 ⚋`）
- HexagramBoard 实时显示当前已掷爻位

## 五、卦象计算引擎

> 核心原则：**规则层不可学习**，所有算法为纯函数，易经知识来自 JSON 数据文件。

### 5.1 引擎模块（12 个文件）

| 模块 | 功能 | 输入 | 输出 |
|------|------|------|------|
| `casting.ts` | 主起卦计算 | `[LineValue×6]` | `HexagramCalculation`（本卦/变卦/动爻/互卦/错卦/综卦/体用/时间）|
| `hexagram-mapping.ts` | 6爻 binary → King Wen 编号 | 6-bit binary | King Wen 1-64（64项映射表）|
| `hexagram-lookup.ts` | 卦象数据查询 | King Wen 编号 | `HexagramData`（卦辞/爻辞/彖传/象传/白话）|
| `hexagram-transforms.ts` | 错卦/综卦 | lines array | King Wen 编号 |
| `trigram-wuxing.ts` | 五行与体用生克 | lines + trigrams | `TiYongRelation`（5种生克方向 + 中文解释）|
| `calendar.ts` | 干支日历 | Date | `TimeContext`（年柱/月建/日辰/旺衰/季节）|
| `najia.ts` | 纳甲系统 | King Wen 编号 + lines + time + category | `NajiaResult`（八宫/世应/六亲/用神/旺衰）|
| `yarrow-stalk.ts` | 大衍筮法 | — | `LineValue` 或 `YarrowCastingDetail`（含三变过程）|
| `duplicate-check.ts` | 一事不二占 | 问题 + 记录列表 | `DuplicateInfo`（24小时内重复检查）|
| `feedback-analytics.ts` | 反馈数据分析 | `DivinationRecord[]` | `AnalyticsReport`（准确率/排行/claims分析）|
| `user-profile.ts` | 用户画像 | userId + records | `UserDivinationProfile`（偏好/应验/insights）|
| `prompt-optimizer.ts` | Prompt 版本管理 | version config | 版本比对 + 指标追踪 |

### 5.2 计算链路（`calculateHexagram`）

```
lines[6,7,8,9,7,8]
    │
    ├─→ linesToBinary()       → linesToKingWen()    → original（本卦编号）
    ├─→ 动爻检测（6→7, 9→8）  → linesToKingWen()    → changed（变卦编号）
    ├─→ 动爻位置              → changingLines[]
    ├─→ calculateMutualHexagram()                    → mutual（互卦编号）
    ├─→ calculateCuoGua()                            → cuoGua（错卦编号）
    ├─→ calculateZongGua()                           → zongGua（综卦编号）
    ├─→ calculateTiYong()                            → tiYong（体用生克关系）
    └─→ getTimeContext()                             → timeContext（时间干支上下文）
```

**注意**：`calculateNajia()` 不在 `calculateHexagram()` 的计算链路中。

### 5.3 动爻取用原则

| 动爻数 | 处理方式 |
|:---:|------|
| 0 | 重点展示本卦卦辞 + 彖传 |
| 1 | 重点展示该爻爻辞 |
| 2-5 | 全部动爻爻辞 + 变卦趋势方向 |
| 6 | 本卦与变卦并重。乾坤用九/用六 |

### 5.4 数据文件

- `src/data/hexagrams.json`：64 卦完整数据（卦辞/彖传/大象辞/爻辞/小象辞 + 白话译文），179KB
- `src/data/najia-bagua.json`：八卦纳甲基础数据（宫/五行/纳支/天干/阴阳）

## 六、AI 解读架构

### 6.1 Double-Call 流程

```
用户问题 + 卦象数据
        │
        ▼
┌─────────────────────────────────┐
│ 第一次调用：结构化推理            │
│ Model: deepseek-v4-flash/pro     │
│ System: 易经占断师（六步断卦法）   │
│ Output: 严格 JSON                │
│ { trend, analysis, conditions,   │
│   timeWindow, answer,            │
│   confidence, claims[] }         │
│ Schema 校验: Zod aiReasoningSchema │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 第二次调用：受控叙事              │
│ System: 易经解读师                │
│ Output: 四段式展示               │
│ 【趋势判断】【核心条件】           │
│ 【时间窗口与建议】【综合判断】      │
│ 容错：失败自动重试 3 次           │
└─────────────────────────────────┘
```

### 6.2 六步断卦法（System Prompt 核心）

```
Step 1 — 定体用：分析上下卦五行，体（我）与用（事）的生克关系
Step 2 — 看旺衰：结合月建/日辰，判断爻的五行旺衰
Step 3 — 查变爻：分析动爻位置、数量及变化意义
Step 4 — 参互卦：互卦看事情的内在因素
Step 5 — 综错观：错卦看对立面，综卦看反转视角
Step 6 — 综合断：引用卦辞爻辞做明确判断
```

### 6.3 Prompt 约束

禁止用语：
- 天意、命中注定、吉人天相、大吉大利、上上大吉

强制要求：
- 必须引用卦辞/爻辞原文出处
- 综合判断必须明确，不含混
- Claims 至少 5 条，覆盖 5 类：trend / condition / timeWindow / advice / answer
- 第一次调用使用 JSON 模式 + Zod 运行时校验

### 6.4 Prompt 数据注入

用户 prompt 包含以下结构化数据块：

1. **卦象数据**：本卦/变卦/互卦/错卦/综卦名、上下卦、体用关系、月建日辰
2. **纳甲信息**（如传入）：卦宫/五行、世应位置、用神、各爻干支+六亲+旺衰
3. **原文引用**：卦辞/爻辞/彖传/小象辞原文 + 白话译文
4. **用户问题**：类别 + 问题原文

### 6.5 模型策略

| 场景 | 模型 | 触发方式 |
|------|------|---------|
| 默认解读 | `deepseek-v4-flash` | 结果页加载自动触发 |
| 深度分析 | `deepseek-v4-pro` | 用户手动点击触发 |

### 6.6 离线降级

`fallback-interpretation.ts`：无 AI 时基于卦象数据生成模板解读
- 从卦象提取卦辞/象辞
- 根据动爻数量生成建议（无动→以静制动，少动→稳步推进，多动→多方考虑，六爻全动→暂时观望）

### 6.7 API 代理

开发环境：Vite proxy `/api/deepseek` → `api.deepseek.com`
生产环境：Vercel serverless function `api/deepseek.ts`

## 七、数据与认证架构

### 7.1 存储分层

| 数据 | 存储位置 | 访问控制 |
|------|---------|---------|
| 古籍数据 | `src/data/*.json` | 只读，构建时打包 |
| 占卜记录 | Supabase PostgreSQL `records` 表 | RLS：用户只能访问自己的 |
| 用户设置 | Supabase `user_settings` 表 | RLS：用户只能访问自己的 |
| API Key | `localStorage` + Supabase 云端同步 | 永不离开浏览器 |
| 数据导出 | JSON 文件下载 | 手动触发 |

### 7.2 数据库 Schema

```sql
records (
  id UUID PK,
  user_id UUID FK → auth.users,
  schema_version INT,
  timestamp TIMESTAMPTZ,
  question TEXT,
  category VARCHAR(20),
  method VARCHAR(10),
  before_divination JSONB,
  hexagram JSONB,        -- { original, changed, changingLines, mutual?, cuoGua?, zongGua?, tiYong?, timeContext? }
  interpretations JSONB, -- InterpretationResult[]
  feedback JSONB,        -- { dueAt, status, detail? }
  duplicate JSONB,
  created_at, updated_at
)

user_settings (
  user_id UUID PK FK → auth.users,
  api_key TEXT,
  preferences JSONB,
  created_at, updated_at
)
```

索引覆盖：user_id, timestamp, category, feedback->>'status', feedback->>'dueAt', schema_version

### 7.3 认证体系

- **Supabase Auth**：用户名 + 密码注册/登录
- **AuthContext**：React Context 全局认证状态
- **ProtectedRoute**：未登录 → 重定向 `/login`
- **API Key**：`localStorage` 存储，支持 `api-key-changed` 事件，可选同步到 Supabase

### 7.4 反馈系统

| 特性 | 实现 |
|------|------|
| 快速反馈 | 三按钮：准/不准/不清楚。无需展开详情 |
| 详细反馈 | 实际结果/满意度(1-5)/耗时/行动/AI影响/claims逐条验证/备注 |
| 提醒时间 | 按类别自动（工作7天/健康14天/其他7天），用户可调整为"稍后提醒"(+3天) |
| 批量反馈 | `FeedbackList.tsx` 模态，逐个处理待反馈记录 |
| 数据持久化 | 更新 Supabase `records.feedback` JSONB 字段 |

## 八、音效与触觉

`src/lib/audio.ts` — Web Audio API 程序化合成，零外部音频文件。

| 音效 | 合成方式 | 时长 | 触发 |
|------|---------|:--:|------|
| 掷钱 | sine 波 1800 → 600Hz 指数降频 | 100ms | 点击铜钱 |
| 单枚落地 | 白噪声 + 800Hz 带通滤波 | 150ms | 单枚铜钱翻转完成 |
| 三次落地 | 三枚错开 0.08s | ~310ms | 三枚铜钱全部完成 |
| 确认爻位 | sine 波 80 → 40Hz 低频 | 200ms | 爻位记录 |

| 触觉 | 实现 | 触发 |
|------|------|------|
| 掷钱振动 | `navigator.vibrate([15, 30, 15])` | 点击铜钱 |
| 确认振动 | `navigator.vibrate(10)` | 爻位确认 |

降级：Web Audio API 不支持 → 静默运行，不报错。

## 十一、关键断裂点

> 以下问题通过 grep 逐链路验证，确认为代码层面真实存在的断裂。

### 断裂 1：纳甲系统完全断开 ⚠️ 影响最大

纳甲引擎（`najia.ts`）是项目中代码量最大、规则最复杂的模块之一（244 行），但存在 **4 个断点**：

```
calculateNajia()         ← 函数存在但从未被调用
    ↓ ❌
calculateHexagram()      ← 不计算纳甲，返回值无 najia 字段
    ↓ ❌
DivinationRecord.hexagram ← 类型定义无 najia 字段，存不了
    ↓ ❌
useAIInterpretation.ts   ← triggerCall() 不传 najia 给 AI
    ↓ ✅（唯一没断的）
DoubleCallInput           ← 有 najia?: NajiaResult 可选字段
    ↓ ✅
prompt-builder.ts         ← 知道如何将纳甲数据格式化为 prompt
```

**实际后果**：八宫卦、世应位置、六亲（父母/兄弟/妻财/官鬼/子孙）、用神定位、各爻旺衰——这些数据 AI 从未收到过。AI 解读质量因此打了折扣。

**验证命令**：

## 十三、架构质量评价

### 做得好的 ✅

- **规则层不可学习贯彻彻底**：`calculateHexagram` 是纯函数，所有易经知识来自 JSON 数据文件，算法与数据完全分离
- **测试哲学一致**：规则引擎严格 TDD，概率用蒙特卡洛验证，Schema 包含向后兼容测试
- **降级设计周到**：无 AI → 模板解读、无 Web Audio → 静默、无障碍模式 → 跳过动画。错误状态都有覆盖
- **Nothing Design 系统执行一致**：CSS 变量命名规范（`--color-nothing-*`），组件复用率高
- **数据版本化**：`schemaVersion` 字段确保未来迁移兼容，导入时校验版本
- **安全性**：Supabase RLS 行级安全，API Key 存 localStorage 永不离开浏览器
- **隐私设计**：用户画像分析在客户端完成，数据不离开本地

### 需要关注的 ⚠️

- **引擎-UI 耦合松散**：引擎很多模块写完就停了，缺少"最后一公里"的 UI 集成。这是最突出的结构性问题
- **类型系统缺口**：`DivinationRecord.hexagram` 缺 `najia` 字段。不会报错因为都是 optional，但语义不完整
- **无 CI/CD**：测试需要手动运行，PR 合并前无自动检查
- **冗余依赖**：`motion` 未使用但仍安装

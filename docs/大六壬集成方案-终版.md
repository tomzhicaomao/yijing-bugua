### 1.2 防误判层设计理念（直接来自 deep research 成败案例）

> **核心原则**：工具不应告诉用户"答案是什么"，而应展示"有哪些可能的解读方向"——所有断语标注"参考性质"。

| 失败根源 | 防误判机制 | 实现位置 |
|---------|-----------|---------|
| 忽略课体 → 直接看神煞 | **课体格局优先展示**：结果页第一区必先展示课体格局 | `LiurenResultView.tsx` |
| 神煞滥用 → 选择性解读 | **矛盾标识**：吉凶神煞冲突时橙色高亮 ⚠️ | `ShenShaConflict.tsx` |
| 死搬毕法赋 | **多解展示**：同一课给出 ≥2 种可能解读方向（乐观/保守/注意） | `LiurenResultView.tsx` |
| 节气错误 → 全盘皆错 | **节气校验**：起课时自动检测是否在节气转换 ±1 天窗口 | `useLiuren.ts` |
| 忽略太岁 | **太岁月建校验**：自动计算太岁与课传刑冲克害 | `防误判层/taiSuiCheck.ts` |
| 主观投射 → 确认偏误 | **反例提示**：用户点选「吉」象时，主动展示相矛盾的象意 | `LiurenResultView.tsx` |

### 4.2 各宗门的精确规则

#### ① 贼克法 (zeke)

```
输入：四课（四组上下神的生克关系）
规则：
  1. 遍历四课，找出所有"下贼上"（下神克上神）
  2. 若有"下贼上"：
     a. 仅一个 → 取下贼上的上神为初传
     b. 多个 → 返回多个候选（交由比用法决疑）
  3. 若无"下贼上"，找所有"上克下"：
     a. 仅一个 → 取上克下的上神为初传
     b. 多个 → 返回多个候选
  4. 既无下贼上也无上克下 → 返回 null

中传 = 初传本身在地盘所乘之神（初传转找天盘）
末传 = 中传本身在地盘所乘之神

输出：{ 初传, 中传, 末传 } 或 null（表示无克，进入比用）
```

#### ② 比用法 (biyong)

```
前提：贼克法返回了多个候选上神（多个下贼上或多个上克下）
规则：取与日干阴阳相同者
  日干为阳（甲丙戊庚壬）→ 取候选中的阳支（子寅辰午申戌）
  日干为阴（乙丁己辛癸）→ 取候选中的阴支（丑卯巳未酉亥）
  若仍有多个符合 → 返回 null（进入涉害法）

输出：{ 初传, 中传, 末传 } 或 null
```

#### ③ 涉害法 (shehai) ⚠️ 最复杂

```
前提：比用法返回了 null（多个候选且多与日干同阴阳）

涉害深度计算方法：
  对每个候选地支，计算其在地盘的涉害深度：
  从该地支在地盘的位置开始，顺时针数十二位
  每遇到一次"受克"（该位地支被候选地支五行所克）就记一次
    
五行相克：
  木克土、土克水、水克火、火克金、金克木

算法伪代码：
  function calculateShehaiDepth(candidateBranch, diPan, tianPan):
    depth = 0
    startIndex = diPan.indexOf(candidateBranch)
    从第 1 位到第 12 位：
      当前地支 = diPan[(startIndex + i) % 12]
      若 candidateBranch 五行克 当前地支五行 → depth++
    返回 depth

规则：
  1. 计算所有候选的涉害深度
  2. 取涉害最深者（最大值）对应的上神为初传
  3. 若深度相同 → 取在地盘上先出现的（四课中先出现者）

输出：{ 初传, 中传, 末传 }
```

#### ④ 遥克法 (yaoke)

```
前提：四课无克（贼克法返回 null）
规则：
  1. 遍历四课，找"日干遥克上神"（即日干克四课的上神）→ 取该上神为初传
  2. 若无，找"上神遥克日干"（即上神克日干）→ 取该上神为初传  
  3. 均无 → null

输出：{ 初传, 中传, 末传 } 或 null
```

#### ⑤ 昴星法 (maoxing)

```
前提：四课全为比和（四课均无克关系）
规则：
  阳日（日干为甲丙戊庚壬）：
    初传 = 酉（昴星）在天盘所临之神（即地盘酉位置的天盘神）
    中传 = 日支的上神（第三课的上神）
    末传 = 日干的上神（第一课的上神）
  
  阴日（日干为乙丁己辛癸）：
    初传 = 酉在地盘所乘之神（即天盘酉位置的地盘神的反向，实际取天盘酉的下神）
    中传 = 日干的上神（第一课的上神）
    末传 = 日支的上神（第三课的上神）

特殊情况：若初传与日干相同（伏吟昴星），特殊处理

输出：{ 初传, 中传, 末传 }
```

#### ⑥ 别责法 (bieze)

```
前提：四课中有三课相同（即两课完全重复）
规则：
  阳日 → 初传 = 日干合神（甲己合→取己寄宫未上神，乙庚合→庚寄宫申上神...）
  阴日 → 初传 = 日支的前三合（如日支丑→丑的前三合为巳酉丑→取巳上神）

  中传 = 末传 = 初传（三传相同）

输出：{ 初传, 中传, 末传 }
```

#### ⑦ 八专法 (bazhuan)

```
前提：干支同位（日干寄宫与日支相同）
规则：
  阳日 → 初传 = 日干上神，在地盘顺数第三位（从日干上神位置开始顺数三）
  阴日 → 初传 = 日支上神（第三课的上神），在天盘逆数三（从日支上神逆数三）

  中传 = 末传 = 初传

输出：{ 初传, 中传, 末传 }
```

#### ⑧ 伏吟法 (fuyin)

```
前提：天地盘完全相同（天盘未旋转/月将=占时）
规则：
  阳日 → 初传 = 日干上神
  阴日 → 初传 = 日支上神

  中传 = 初传的刑神（子刑卯、卯刑子、寅刑巳、巳刑申、申刑寅、
                         丑刑戌、戌刑未、未刑丑、辰午酉亥自刑）
  末传 = 中传的刑神

特殊情况：若中传自刑
  → 阳日取日支上神为中传，阴日取日干上神为中传
  → 若仍自刑，取日马为中传

输出：{ 初传, 中传, 末传 }
```

#### ⑨ 返吟法 (fanyin)

```
前提：天盘与地盘的十二地支完全相冲（每日支冲位）
规则：
  1. 找出四课中克日干的课（上神克日干）→ 取该上神为初传
  2. 若有多个 → 用比用法
  3. 若无克日干 → 取驿马（日马）为初传

  中传 = 初传本身在地盘所乘之神
  末传 = 中传本身在地盘所乘之神

输出：{ 初传, 中传, 末传 }
```

## 五、防误判机制

> **设计来源**：Deep Research 中发现的 5 种失败模式，每种对应一个防误判机制。

### 5.1 课体格局优先展示

```
失败模式：占者忽略课体格局，直接看神煞 → 与忽略大框架的取舍失误

实现：
  LiurenResultView 第一区域必须是「课体格局」
  
  ┌─────────────────────────────┐
  │  📋 课体格局：重审（吉）       │
  │  ───────────────────────    │
  │  重审者，事有反复，须再三审视   │
  │  凡事宜缓不宜急，静观其变。    │
  └─────────────────────────────┘
  
  位置固定在结果页最顶部，优先于三传、天将、神煞。
```

### 5.2 矛盾标识

```
失败模式：吉凶神煞冲突时选择性关注吉煞 → 确认偏误

实现逻辑：
  遍历 shenSha 数组，找出以下矛盾对：
  • 吉煞 + 凶煞 临同一地支
  • 青龙（吉）临空亡
  • 天喜（吉）+ 飞廉（凶）同时入传

  输出 warnings 数组，UI 中橙色高亮：
  ⚠️ 天喜（吉）与飞廉（凶）临同一课位，吉凶交战，事有反复
  ⚠️ 青龙临空亡，吉力不足
```

### 5.3 太岁/月建校验

```
失败模式：邵彦和案例——忽略太岁与课传的冲突

实现逻辑：
  太岁 = 当年地支（如 2024 甲辰年 → 太岁 = 辰）
  月建 = 当月地支
  
  检查：
  • 太岁刑冲克日干 → 警告
  • 太岁刑冲克三传初传 → 警告
  • 太岁生合日干 → 正面提示
  
  示例输出：
  ⚠️ 太岁辰土克日干壬水，本年大环境对所占之事有压制
  ✅ 太岁辰土生日干甲木，本年大环境有利
```

### 5.4 节气边界校验

```
失败模式：节气交接时月将判断错误 → 全盘错误

实现逻辑：
  getYueJiang 函数内部：
    date 与节气表中的前后节气对比
    若 |date - 节气时间| < 24小时 → 设置 warning
    
  示例输出：
  ⚠️ 当前时间距惊蛰仅 12 小时，月将可能在「亥」与「戌」之间
  请确认时间无误后重新起课
```

### 5.5 反例提示

```
失败模式：占者选择性关注符合预期的象意
```

## 六、AI 解读层

### 6.1 六步断课法 Prompt 框架

```markdown
# System Prompt: 大六壬解课助手

你是一位精通大六壬的占卜解课师。你遵循"六步断课法"进行解课。
你给出的解读永远是"参考性质"，绝不宣称绝对答案。

## 六步断课法

### 第一步：定课体
根据课式中的格局（元首/重审/贼克/涉害...），判断课体吉凶大方向。
参考规则：
- 元首：一上克下，万事顺利，宜主动
- 重审：一下贼上，事有反复，宜谨慎
- 知一：比用取一，事有分别
- 涉害：多克取深，事多艰难
- 遥克：遥相克，事不直接
- 昴星：四课无克，事多晦暗
- 伏吟：天地相同，事态停滞
- 返吟：天地相冲，事态反复

### 第二步：审三传
看初传（事始）、中传（事中）、末传（事终）：
- 三传递生：事越发展越好
- 三传递克：事越发展越难
- 末传生初传：终归有好
- 初传克末传：始吉终凶

### 第三步：察天将
十二天将在三传中的分布：
- 贵人、青龙、太常、六合为吉将
- 螣蛇、白虎、玄武为凶将
- 朱雀主口舌、天空主虚诈
- 看天将是否入传，是否得位

### 第四步：辨六亲
三传的六亲关系（父母/兄弟/妻财/官鬼/子孙）：
- 占官喜官鬼，占财喜妻财，占学喜父母，占孕喜子孙
- 看六亲是否与所占之事匹配

### 第五步：参神煞
神煞作为辅助参考：
- 驿马主动、桃花主情、天喜主庆、飞廉主灾
- 神煞不可喧宾夺主，必须放在课体和三传之后考虑

### 第六步：综合断课
综合以上五步，给出：
1. 总体吉凶判断（带置信度说明）
2. 关键象意解读
3. 应期推算（如适用）
4. ⚠️ 反例提示（必须包含）
```

#### 16.2.3 跨请求 LRU 缓存（节流重复计算）

```typescript
// src/engine/liuren/cache.ts
// 用户在同一会话内频繁切换时间时生效

interface CacheEntry {
  pan: LiurenPan;
  ts: number;
}

class LiurenCache {
  private cache = new Map<string, CacheEntry>();
  private maxSize = 200;
  private ttl = 10 * 60 * 1000; // 10 分钟
  
  get(key: string): LiurenPan | null {
    const entry = this.cache.get(key);
    if (!entry) return null;
    if (Date.now() - entry.ts > this.ttl) {
      this.cache.delete(key);
      return null;
    }
    // LRU: 移到末尾
    this.cache.delete(key);
    this.cache.set(key, entry);
    return entry.pan;
  }
  
  set(key: string, pan: LiurenPan): void {
    // 淘汰最旧
    if (this.cache.size >= this.maxSize) {
      const oldest = this.cache.keys().next().value;
      if (oldest) this.cache.delete(oldest);
    }
    this.cache.set(key, { pan, ts: Date.now() });
  }
  
  // 缓存键 = 日期时间 + 活时地支（正时为 null）
  static key(date: DateTime, shiZhi?: Branch): string {
    return `${date.toISO()}:${shiZhi ?? 'zheng'}`;
  }
}

export const liurenCache = new LiurenCache();
```

```typescript
  const startIdx = BRANCH_INDEX[branch];
  const myWuXing = BRANCH_TO_WUXING[branch];
  let depth = 0;
  
  // 使用数组索引运算，避免 indexOf 和字符串比较
  for (let i = 1; i <= 12; i++) {
    const targetIdx = (startIdx + i) % 12;
    const targetWuXing = branchWuXing[targetIdx];
    if (keMatrix[myWuXing][targetWuXing]) {
      depth++;
    }
  }
  return depth;
}
```

### 16.5 AI 调用优化

```typescript
// src/ai/liuren-call.ts

const AI_TIMEOUT = 25_000;        // 单次调用超时
const AI_RETRY_MAX = 2;           // 最多重试 2 次
const AI_RETRY_BACKOFF = [1000, 3000];  // 重试间隔：1s, 3s

async function callWithRetry<T>(
  fn: () => Promise<T>,
  timeout: number,
  retries: number
): Promise<T> {
  for (let attempt = 0; attempt <= retries; attempt++) {
    try {
      const result = await Promise.race([
        fn(),
        new Promise<never>((_, reject) =>
          setTimeout(() => reject(new Error('AI timeout')), timeout)
        ),
      ]);
      return result;
    } catch (err) {
      if (attempt === retries) throw err;
      // 仅网络错误重试，4xx/5xx 不重试
      if (err instanceof TypeError || err.message === 'AI timeout') {
        await new Promise(r => setTimeout(r, AI_RETRY_BACKOFF[attempt]));
        continue;
      }
      throw err;
    }
  }
  throw new Error('Unreachable');
}
```

### 18.2 错误处理代码模式

```typescript
// src/lib/result.ts — Rust 风格 Result 类型

type Result<T, E = string> = 
  | { ok: true; value: T }
  | { ok: false; error: E; errorCode: string; userMessage: string };

// 使用示例：每个可能失败的操作都返回 Result
async function saveRecord(record: NewRecord): Promise<Result<DivinationRecord>> {
  try {
    const { data, error } = await supabase
      .from('records')
      .insert(record)
      .select()
      .single();
    
    if (error) throw error;
    return { ok: true, value: data };
  } catch (err) {
    // 本地缓存兜底
    localStorage.setItem(`pending-record-${Date.now()}`, JSON.stringify(record));
    
    return {
      ok: false,
      error: String(err),
      errorCode: 'DB_WRITE_FAILED',
      userMessage: '保存失败，数据已缓存在本地，网络恢复后自动同步',
    };
  }
}

// 后台同步
```

```typescript
// src/lib/validation.ts

const QUESTION_MAX_LENGTH = 200;
const QUESTION_MIN_LENGTH = 1;

function validateQuestion(input: string): Result<string> {
  const trimmed = input.trim();
  
  if (trimmed.length < QUESTION_MIN_LENGTH) {
    return { ok: false, error: '问题不能为空', errorCode: 'QUESTION_EMPTY', userMessage: '请输入您想问的问题' };
  }
  
  if (trimmed.length > QUESTION_MAX_LENGTH) {
    return { ok: false, error: `问题超过${QUESTION_MAX_LENGTH}字`, errorCode: 'QUESTION_TOO_LONG', userMessage: `问题请控制在${QUESTION_MAX_LENGTH}字以内` };
  }
  
  // 拒绝纯空白字符
  if (/^\s+$/.test(trimmed)) {
    return { ok: false, error: '问题不能全为空白', errorCode: 'QUESTION_WHITESPACE', userMessage: '请输入有效的问题' };
  }
  
  return { ok: true, value: trimmed };
}

function validateDateTime(date: DateTime): Result<DateTime> {
  const ts = date.toMillis();
  const MIN_TS = DateTime.fromISO('1900-01-01').toMillis();
  const MAX_TS = DateTime.fromISO('2100-12-31').toMillis();
  
  if (ts < MIN_TS || ts > MAX_TS) {
    return { ok: false, error: '日期超出范围', errorCode: 'DATE_OUT_OF_RANGE', userMessage: '日期超出支持范围（1900-2100 年）' };
  }
  
  return { ok: true, value: date };
}
```

### 19.3 频率限制

```typescript
// src/hooks/useRateLimit.ts

interface RateLimitConfig {
  windowMs: number;     // 时间窗口
  maxRequests: number;  // 最大请求数
}

function useRateLimit(config: RateLimitConfig) {
  const timestamps = useRef<number[]>([]);
  
  const checkAndRecord = useCallback((): { allowed: boolean; retryAfterMs: number } => {
    const now = Date.now();
    const windowStart = now - config.windowMs;
    
    // 清理过期记录
    timestamps.current = timestamps.current.filter(t => t > windowStart);
    
    if (timestamps.current.length >= config.maxRequests) {
      const oldestInWindow = timestamps.current[0];
      const retryAfter = oldestInWindow + config.windowMs - now;
      return { allowed: false, retryAfterMs: Math.max(retryAfter, 1000) };
    }
    
    timestamps.current.push(now);
    return { allowed: true, retryAfterMs: 0 };
  }, [config.windowMs, config.maxRequests]);
  
  return { checkAndRecord };
}

// 使用
// 每用户每分钟最多 5 次起课
const rateLimit = useRateLimit({ windowMs: 60_000, maxRequests: 5 });
```
